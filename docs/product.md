## 1) Описание продукта

### Кратко (что за сервис)
Мобильное приложение для **исполнителей/владельцев спецтехники и диспетчеров**, которое в реальном времени доставляет **заявки на аренду техники (лиды)** по выбранным категориям и регионам, позволяет **быстро оценить заявку**, **открыть контакты заказчика** и **закрыть сделку**. Плюс — отдельный сценарий для заказчика: **разместить заказ** и получить отклики.

### Для кого (ЦА)
- **Владельцы 1–10 единиц спецтехники** (экскаваторы, самосвалы, автовышки и т.д.), часто без выделенного отдела продаж.
- **Диспетчеры/агрегаторы/парки техники** (10–200 единиц), где важно распределять лиды, контролировать конверсию, аналитику.
- (Опционально во 2-й роли) **Заказчики**: прорабы, снабженцы, ПТО, небольшие подрядчики — им нужен быстрый заказ.

### Проблема
- Лиды приходят хаотично (мессенджеры/звонки), **теряются** и **не обрабатываются вовремя**.
- Невозможно быстро отфильтровать “мусорные” заявки и понять **цена/локация/срок/условия**.
- Контакты заказчика — ключевой актив, нужно **защитить монетизацию** (free → paid) без ощущения “обмана”.
- В B2B важны **скорость реакции**, **прозрачность статуса**, **управление командой** (если диспетчер + водители).

### Ценность
- **Исполнителю**: больше релевантных лидов, быстрее обработка, контроль стоимости контакта и конверсии.
- **Диспетчеру/парку**: распределение лидов, статистика, дисциплина обработки.
- **Заказчику**: быстрый заказ без обзвона десятков телефонов.

## 2) Основные фичи

### 2.1 Фичи “как в боте” (переносим 1:1)
- **Подписки на категории техники**: выбор категорий + регион/радиус/город (обязательно для качества лидов).
- **Поток заявок (лента лидов)**: новые, в работе, закрытые.
- **Просмотр контактов заказчика**: открытие телефона/Telegram/WhatsApp по нажатию.
- **Лимиты и подписки**: 3 бесплатных просмотра → дальше paywall.
- **Размещение заказа** (роль заказчика или форма “передать заказ”): создать заявку с параметрами.

### 2.2 Улучшения, которые “просятся” именно в мобилке (B2B-ориентир)
1) **Квалификация заявки до открытия контакта**
- Показать до paywall: **категория, локация, время, смена/часы, объём/тип работ, требования, ориентир бюджета** (если есть).
- **Скоринг релевантности**: “Подходит по вашим категориям/радиусу/доступности”.

2) **Действия в 1 тап после открытия**
- Кнопки: **Позвонить**, **WhatsApp**, **Telegram**, **Скопировать номер**, **Построить маршрут**.
- Таймер “время реакции” (в B2B это KPI).

3) **Статусы лида + причины**
- “Новый” → “В работе” → “Договорились”/“Не подошло”.
- Причины отказа: “далеко”, “нет техники”, “цена”, “сроки”, “дубль”, “спам”. Это кормит аналитику качества.

4) **Управление доступностью (availability)**
- “Техника свободна с…”, “занято до…”, “принимаю только смены”, “ночные работы — да/нет”.
- Фильтрация лидов по доступности снижает “пустые открытия контактов”.

5) **Командный режим (для парков)**
- Роли: **владелец/админ**, **диспетчер**, **менеджер**, (опционально) **водитель**.
- Назначение лида сотруднику, внутренние заметки.

6) **Антидубли и защита монетизации**
- Если контакт уже открывали по этой заявке — повторно не списываем (или явно показываем правило).
- Если заявка “одна и та же” (дубликаты от заказчика) — объединяем.

7) **Качество лидов**
- Рейтинг заказчика (простая метрика): “подтверждённый номер”, “история заказов”, “частота отмен”.
- Отметка “проверено модерацией” (позже).

8) **Офферы/отклики (опционально, но сильная фича)**
- Вместо “купить контакт” можно дать модель “Откликнуться”: отправить цену/условия, заказчик выбирает.
- Это сложнее, но повышает ценность и снижает ощущение “платишь за номер”.

## 3) User Flow

### 3.1 Первый вход (онбординг)
1) Экран “Кто вы?”: **Исполнитель / Парк / Заказчик** (можно начать только с “Исполнитель”, заказчика открыть позже).
2) Вход по **телефону (SMS/flash-call)** или Telegram OAuth (если реально удобно).
3) Мини-онбординг (3–4 карточки): “Выбираете категории → получаете заявки → открываете контакты → закрываете сделку”.
4) Экран “Настройте получение”: **город/радиус**, часы уведомлений, язык.

### 3.2 Выбор категорий
1) Каталог категорий → выбор.
2) Для каждой категории (опционально): подкатегории/параметры (тоннаж, стрела, объём ковша).
3) Подтверждение: “Вы будете получать заявки по …”.

### 3.3 Получение заявок
1) Push “Новая заявка: Экскаватор, 5 км от вас”.
2) Переход в карточку заявки: краткий summary + детали + CTA.

### 3.4 Просмотр контакта
1) Нажали “Открыть контакт”.
2) Если есть free-лимит: показываем “Это будет 1 из 3 бесплатных”.
3) Если лимит исчерпан: paywall → выбор тарифа → оплата → возврат в заявку → контакт открыт.

### 3.5 Покупка подписки
1) Paywall с тарифами (месяц/квартал/год) + чётко “что включено”.
2) Оплата (App Store / Google Play) + подтверждение.
3) Синхронизация подписки с backend, активация прав.
4) Возврат в контекст (ту же заявку), без “потери” пользователя.

## 4) Структура экранов (IA)

### 4.1 Главный экран (Заявки)
- Вкладки: **Новые / В работе / Архив**
- Фильтры: категория, расстояние/город, дата/время, “с контактами открыто/не открыто”.
- Карточка заявки: категория, локация, время, ключевые параметры, бейдж “NEW”, бейдж “Подходит вам”.

### 4.2 Категории
- Список категорий (поиск + избранное).
- Настройки подписки: регион/радиус, параметры техники, “получать ночью” и т.п.
- Экран “Мои категории” (быстрый доступ к активным).

### 4.3 Мои подписки (и доступ)
- Текущий план, дата окончания, автопродление.
- Счётчик: “Осталось бесплатных просмотров: 1/3” (пока free).
- Кнопка “Улучшить тариф”.

### 4.4 История
- Журнал открытых контактов (для прозрачности списаний).
- История лидов со статусами и причинами.
- Экспорт (позже): CSV/почта.

### 4.5 Размещение заказа
- Форма: категория, локация (точка/адрес), дата/время, длительность, параметры, комментарий, контактный телефон, способ связи.
- Подтверждение + “ваш заказ отправлен исполнителям по …”.

### 4.6 Тарифы (Paywall)
- Планы + сравнение.
- Чёткие правила: что считается “просмотром контакта”, повторные списания, возвраты (если применимо).

### 4.7 Профиль
- Данные компании/ИП, ИНН (опционально), город.
- Режим работы уведомлений.
- Роли/сотрудники (если парк).
- Поддержка, документы, политика.

## 5) UX-логика (ключевые механики)

### 5.1 Как показываются заявки
- **Лента = приоритет “срочно + близко + по вашим категориям”**.
- В карточке до открытия контакта показываем максимум, чтобы решение “стоит ли платить/тратить просмотр” было честным:
  - **что нужно** (категория/параметры),
  - **где** (город + примерная зона/километры; точный адрес можно скрывать до контакта, но не всегда),
  - **когда** (сегодня/завтра, время),
  - **объём/срок**,
  - **условия** (нал/безнал, пропуск, работа в ограничениях).
- CTA:
  - “Открыть контакт” (платный/лимитный),
  - “Сохранить/В работу” (бесплатно),
  - “Не подходит” (с причиной).

### 5.2 Как ограничивается доступ к контактам
- Сущность “просмотр контакта” = **единоразовое открытие для конкретной заявки**.
- До открытия: контакты замаскированы.
- После открытия: контакты доступны в заявке + в “Истории открытий”.
- Если пользователь пытается открыть контакт при исчерпанном лимите:
  - показываем paywall,
  - после покупки возвращаем **в ту же заявку** и сразу открываем контакт (минимум трения).

### 5.3 Как работает free → paid
- Free лимит показываем **в интерфейсе заранее**:
  - бейдж вверху ленты “Бесплатных просмотров осталось: 2”.
- Первые 3 открытия сопровождаем микро-подсказкой:
  - 1-е: “У вас 3 бесплатных просмотра”
  - 3-е: “Это последний бесплатный”
- После исчерпания:
  - в карточке заявки CTA меняется на “Открыть контакт по подписке”
  - paywall объясняет: “Контакты — платная часть, чтобы защитить качество и монетизацию”.

## 6) Монетизация

### 6.1 Модель
- Основная: **подписка** (лучше для B2B, прогнозируемость).
- Дополнительно (опционально): **пакеты просмотров** (например, 10/30/100) для тех, кто не любит подписки.

### 6.2 Ограничения (прозрачные правила)
- **3 бесплатных открытия контактов** на аккаунт (или на месяц — решите стратегию; для B2B чаще “на аккаунт”, чтобы не абьюзили).
- Открытие контакта по заявке списывается 1 раз, повторный просмотр — без списания.
- Возможность ограничения на “команду”: подписка на компанию/аккаунт или на пользователя (для парков лучше “на компанию”).

### 6.3 Варианты тарифов (пример, логика)
- **Starter**: X открытий/мес или безлимит контактов, но ограничение по категориям (например, до 2 категорий).
- **Pro**: больше категорий + приоритетные уведомления + командные функции базовые.
- **Park/Team**: мультипользовательский доступ, назначение лидов, аналитика, SLA поддержки.

## 7) API контракт (backend-ready)

Ниже — минимальный контракт, чтобы мобилка была самостоятельной и быстро повторила поведение бота.

### 7.1 Авторизация и пользователь
- `POST /v1/auth/otp/request` — запрос OTP (телефон)
- `POST /v1/auth/otp/verify` — подтверждение, выдача `access_token`, `refresh_token`
- `POST /v1/auth/refresh`
- `GET /v1/me` — профиль + текущие права (лимит, подписка, роли)
- `PATCH /v1/me` — обновление профиля/настроек (город, уведомления)

**User (пример структуры)**
- `id`, `phone`, `name`, `role` (contractor/park/customer)
- `company_name` (опционально)
- `region_id`, `city_id`
- `free_contact_views_left`
- `subscription` (nullable, объект)

### 7.2 Категории и подписки на категории (не платежные)
- `GET /v1/categories` — список категорий/подкатегорий
- `GET /v1/subscriptions/categories` — мои подписки на категории
- `POST /v1/subscriptions/categories` — добавить категорию + настройки (радиус, параметры)
- `PATCH /v1/subscriptions/categories/{id}`
- `DELETE /v1/subscriptions/categories/{id}`

**CategorySubscription**
- `id`, `category_id`
- `geo` (город/радиус/координаты)
- `params` (ключ-значение по технике)
- `enabled`

### 7.3 Лиды (заявки) и лента
- `GET /v1/leads?status=new&category_id=&cursor=` — лента (пагинация cursor)
- `GET /v1/leads/{id}` — детали заявки
- `POST /v1/leads/{id}/status` — смена статуса (in_progress/won/lost) + причина
- `POST /v1/leads/{id}/contact/preview` — “что будет при открытии” (для UX: покажет, можно ли бесплатно/платно)
- `POST /v1/leads/{id}/contact/reveal` — открыть контакт (списание free или проверка подписки)
- `GET /v1/contact-views` — история открытий

**Lead**
- `id`, `category_id`, `category_title`
- `created_at`, `scheduled_at`, `duration_hours`
- `location`:
  - `city`, `lat`, `lng`, `distance_km` (от пользователя), `address_hint` (без точного адреса, если надо)
- `requirements` (текст + структурные поля)
- `budget` (nullable: `amount`, `currency`, `type` hourly/shift/fixed)
- `customer` (до reveal: `name_masked`, `rating_flags`; после reveal: `phone`, `messengers`)
- `contact_state`: `hidden | revealed`
- `status`: `new | in_progress | archived | won | lost`
- `is_duplicate` (optional)

**ContactRevealResponse**
- `lead_id`
- `contact` { `phone`, `telegram`, `whatsapp` }
- `billing` { `type`: free|subscription|pack, `remaining_free`: number }

### 7.4 Заказ (размещение)
- `POST /v1/orders` — создать заказ (заказчик)
- `GET /v1/orders/{id}`
- (Если заказчик в MVP не нужен — оставляем только создание, без кабинета)

**OrderCreate**
- `category_id`
- `location` (адрес/координаты)
- `scheduled_at`, `duration_hours` или `shift_count`
- `details` (поля по категории)
- `contact_phone`, `preferred_channel`

### 7.5 Подписка (App Store / Google Play)
- `GET /v1/plans` — список тарифов (для paywall)
- `POST /v1/billing/purchase` — отправка purchase token/receipt на backend
- `GET /v1/billing/subscription` — статус подписки
- `POST /v1/billing/webhook/*` — серверные вебхуки стора (на backend)

## 8) Технический стек

### 8.1 Что выбрать: React Native / Flutter / Native
**Рекомендация: React Native** (если у вас уже JS/TS-экосистема или есть веб-команда).
- Быстрое достижение parity с ботом, быстрые итерации UX.
- Хороший набор SDK для IAP, push, deep links.
- Дешевле в поддержке, чем 2 native-команды.

**Flutter** — альтернатива, если хотите максимально консистентный UI и сильную производительность на списках/анимациях.  
**Native (Swift/Kotlin)** — оправдано, если ожидаете очень сложные нативные интеграции/офлайн/карты/флит-менеджмент, но для MVP обычно избыточно.

### 8.2 Архитектура приложения (практично под B2B)
- **Clean Architecture / Feature-Sliced** по фичам: `auth`, `leads`, `categorySubs`, `billing`, `profile`, `orders`.
- Слой данных: API client + DTO/Mapper → доменные модели.
- **Offline-first для ленты** (кэш последней ленты + статусы), чтобы в поле работало стабильно.
- Аналитика событий (обязательно для монетизации): `lead_open`, `contact_reveal_attempt`, `paywall_view`, `purchase_success`, `lead_status_change`.

### 8.3 Пуши
- FCM (Android) + APNs (iOS) через единый провайдер (например, Firebase).
- Типы пушей:
  - “Новая заявка”
  - “Срочная заявка рядом”
  - “Подписка скоро закончится”
- Deep link прямо в `LeadDetails`.
- Серверная логика “quiet hours” + локальные настройки.
